# 웹 대시보드 설명

## 목적

이 대시보드는 같은 검색 조건을 MongoDB direct 경로와 Mini Redis cache 경로에 동시에 보내고,
두 경로의 응답 결과와 성능 차이를 한 화면에서 비교하기 위한 UI입니다.

## 현재 비교 기준

- Winner 판정 기준은 `server latency`입니다.
- 각 Lane에는 `브라우저 체감`과 `서버 latency`가 모두 표시됩니다.
- 상단 Winner 영역과 상태 메시지는 서버 레이턴시 기준 결과를 설명합니다.

## 화면 구성

### 1. Hero

- 대시보드 제목
- 현재 Redis backend
- 현재 API target

### 2. Race Setup

- 검색어 입력
- 지역 선택
- 카테고리 선택
- `동시 비교 실행` 버튼
- `메트릭 새로고침` 버튼

사용자가 조건을 입력하고 실행 버튼을 누르면 MongoDB와 Redis 요청이 동시에 시작됩니다.

### 3. Winner Panel

- 승자 표시
- 두 경로의 `server latency` 차이 요약
- 진행 바

이 영역은 두 Lane 응답이 모두 끝난 뒤 갱신됩니다.

### 4. MongoDB Direct Lane

- 브라우저 체감
- 서버 latency
- 결과 수
- cache status
- trace 정보
- 검색 결과 카드 목록

### 5. Mini Redis Cache Lane

- 브라우저 체감
- 서버 latency
- 결과 수
- cache status
- trace 정보
- 검색 결과 카드 목록

같은 검색 조건을 반복 실행하면 Redis Lane에서 `miss-fill` 이후 `hit` 상태를 확인할 수 있습니다.

### 6. Service Snapshot

- request count
- error count
- hit ratio
- req/sec
- avg latency
- cpu
- tx / rx

## 동작 흐름

1. 정적 페이지가 먼저 로드됩니다.
2. 초기화 단계에서 설정 정보와 메트릭을 비동기로 불러옵니다.
3. 사용자가 `동시 비교 실행`을 누르면 MongoDB와 Redis 요청이 동시에 전송됩니다.
4. 각 Lane은 응답이 도착하는 즉시 개별적으로 갱신됩니다.
5. 두 Lane이 모두 끝나면 Winner가 `server latency` 기준으로 계산됩니다.

## 관련 코드

- `static/index.html`: 대시보드 화면 구조
- `static/app.js`: 비교 실행, Lane 렌더링, Winner 계산
- `demo_benchmark/routes/api.py`: 검색 및 메트릭 API
- `demo_benchmark/service.py`: Mongo direct / Redis cache 비교 로직
