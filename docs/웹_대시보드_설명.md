# 웹 대시보드 설명

## 목적

이 웹 대시보드는 1번 팀원이 발표와 QA를 위해 사용하는 데모 플랫폼이다. 핵심 목적은 아래 두 가지다.

- 같은 key 기준으로 `MongoDB 직접 조회`와 `Mini Redis 경유 조회`의 차이를 바로 보여주기
- Mini Redis가 정상 동작하는지와 엣지 케이스에서 어떤 결과가 나오는지를 시각적으로 검증하기

이 저장소는 Redis 서버 자체를 구현하지 않고, 2~4번 팀원이 만든 Mini Redis 프로세스를 소비하는 프론트엔드/백엔드 데모 계층을 담당한다.

## 화면 구성

대시보드는 카드형 발표 화면이 아니라, 운영 콘솔에 가까운 관리자 페이지 레이아웃으로 구성된다. 색상은 강한 네온 계열 대신 중성적인 밝은 톤을 사용하고, 각 섹션은 명확한 구획선과 헤더 배지로 분리한다.

### 1. Top Bar

- 현재 세션의 핵심 상태를 상단에 고정해서 보여준다.
- 표시 항목
  - default key
  - default benchmark iterations
  - redis backend
  - api target
- 보조 설명 문구를 추가해 대시보드의 전체 목적을 짧게 안내한다.
- 목적
  - 지금 어떤 환경으로 데모 중인지 즉시 파악하게 한다.
  - 로컬 Docker가 아니라 공유 서버 API를 프록시하는 상태인지 바로 구분하게 한다.

### 2. Left Control Panel

- 사용자가 demo key와 benchmark iterations를 직접 입력할 수 있다.
- 주요 액션 버튼
  - `Run Mongo Direct`
  - `Run Redis Cache`
  - `Run QA Suite`
  - `Run Benchmark`
  - `Refresh Metrics`
- `Run Mongo Direct`와 `Run Redis Cache` 중 어느 버튼을 눌러도 비교 패널에서는 두 경로 결과를 함께 갱신한다.
- 버튼 라벨은 사용자가 어떤 비교를 하려는지 드러내기 위한 것이고, 실제 동작은 direct/cache 동시 비교 실행이다.
- 우측 메인 화면을 어떤 순서로 시연할지 실행 순서도 함께 보여준다.
- 섹션 자체를 별도 패널로 분리해 메인 결과 영역과 시각적으로 명확히 구분한다.

목적:

- 발표용 소개 카드가 아니라, 실제 운영자가 여러 점검 액션을 순서대로 실행하는 도구처럼 보이도록 한다.

### 3. Service Snapshot 패널

- 현재 앱 기준 메트릭을 상단 그리드로 표시한다.
- 섹션 헤더 하단에 구분선을 두어 아래의 결과 영역과 분리한다.
- 표시 지표
  - request count
  - error count
  - cache hit ratio
  - requests/sec
  - avg latency
  - RSS used memory
  - active connections
  - CPU
  - network tx/rx

목적:

- 조회와 QA를 실행한 직후 상태가 어떻게 변했는지 운영 수치 관점에서 바로 읽을 수 있게 한다.

### 4. Lookup Result Inspectors

- 같은 key를 두 경로로 비교한다.
- `MongoDB direct`
  - MongoDB에서 문서를 직접 조회한다.
- `Mini Redis cache`
  - 먼저 Redis에서 key를 조회한다.
  - cache miss면 MongoDB에서 읽고 Redis에 채운 뒤 반환한다.
- 비교 패널은 단일 경로만 갱신하지 않고, direct와 cache를 항상 한 번에 다시 그린다.
- 각 카드에 표시되는 정보
  - latency
  - 조회 source
  - payload JSON
- 결과 패널은 흰색 배경과 내부 코드 박스로 나뉘어 다른 섹션과 경계가 분명하게 보이도록 구성한다.

목적:

- 두 결과를 나란히 두되, 관리자 페이지처럼 JSON inspection 중심으로 보여준다.

### 5. Benchmark 패널

- 같은 key를 기준으로 Mongo direct path와 Redis cache path를 반복 측정한다.
- Redis cache path는 1회 warm-up miss 후 hit 중심으로 측정한다.
- 표시 지표
  - avg
  - min
  - max
  - p95
  - requests/sec
  - speedup
  - artifact path

목적:

- 성능 비교를 실행 결과 요약 패널과 metric 카드 조합으로 보여준다.
- History 패널과 2열 구성을 유지하되 카드 외곽선과 여백을 늘려 서로 섞여 보이지 않게 한다.

### 6. Metrics History 패널

- 최근 snapshot을 표로 보여준다.
- 시간대별로 `req/sec`, `avg latency`, `hit ratio`, `errors`를 비교한다.

목적:

- 현재 스냅샷만 보는 것이 아니라 직전 실행들과의 변화 흐름도 추적하게 한다.

### 7. QA 패널

- Mini Redis가 정상 동작하는지와 edge case에서 어떻게 반응하는지 검증한다.
- 각 항목은 `pass/fail`, `latency`, `expected`, `actual`, `failure reason`을 포함한다.
- 카드 나열 대신 테이블 형태로 재구성해 여러 시나리오를 한 번에 비교할 수 있게 했다.
- 테이블은 별도 래퍼 박스 안에 넣어 다른 패널과 시각적으로 분리한다.

현재 포함된 시나리오:

- `PING`
- `SET` 후 `GET`
- overwrite
- missing key lookup
- multiple key `DEL`
- `EXPIRE` lazy expiration
- `QUIT`
- `EXIT`
- unknown command
- wrong arity
- malformed RESP

목적:

- QA를 운영 점검표처럼 읽을 수 있게 하여, 어떤 시나리오가 실패했는지 바로 찾게 한다.

## 백엔드 연결 구조

웹 대시보드는 아래 계층을 통해 동작한다.

- `demo_benchmark/app.py`
  - WSGI 앱과 API 라우팅
  - `UPSTREAM_API_BASE_URL` 설정 시 공유 서버 API 프록시도 담당
- `docker-compose.yml`
  - 테스트 용도 로컬 MongoDB + 웹 컨테이너 실행용
- `demo_benchmark/service.py`
  - Mongo direct, Redis cache, benchmark, QA 실행 orchestration
- `demo_benchmark/mongo_backend.py`
  - MongoDB 문서 조회/적재
- `demo_benchmark/mini_redis.py`
  - Mini Redis TCP client 및 stub client
- `demo_benchmark/qa.py`
  - QA 시나리오 실행
- `demo_benchmark/metrics.py`
  - monitoring snapshot 및 history 수집

## API 요약

- `GET /api/lookup?key=<key>&mode=mongo|cache`
- `POST /api/qa/run`
- `POST /api/benchmark`
- `GET /api/metrics/current`
- `GET /api/metrics/history`
- `GET /api/config`

모든 응답은 JSON이다.

### 공유 서버 프록시 모드

- `UPSTREAM_API_BASE_URL`이 설정되면 대시보드의 `/api/*` 요청은 로컬 MongoDB/Redis 처리 대신 해당 공유 서버로 프록시된다.
- 이때 상단 Top Bar의 `API Target`에 현재 연결 대상이 표시된다.
- `/api/config` 응답에는 `api_target`, `api_mode`가 추가되어 현재 로컬 모드인지 프록시 모드인지 구분할 수 있다.

## UI 수정 규칙

앞으로 웹 UI 또는 사용자에게 보이는 동작을 수정할 때는 반드시 이 문서를 함께 업데이트한다.

업데이트 대상 예시:

- 패널 추가/삭제
- 버튼/입력 폼 변경
- 카드/표/그래프 구성 변경
- 콘솔 레이아웃 또는 관리자 도구 흐름 변경
- API와 연결되는 화면 동작 변경
- 발표 흐름에 영향을 주는 시각화 변경

최소 반영 기준:

- 변경된 화면 섹션 설명
- 사용자 액션 흐름
- 표시 지표 또는 결과 형식
- 발표 시 어떤 메시지를 전달하는지
