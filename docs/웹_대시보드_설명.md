# 웹 대시보드 설명

## 목적

현재 대시보드는 `MongoDB direct` 와 `Mini Redis cache` 를 같은 검색 조건으로 동시에 호출해,
어느 쪽이 먼저 화면에 도착하는지 직관적으로 보여주는 데 집중한다.

핵심 메시지는 두 가지다.

- 같은 요청을 비동기로 동시에 보냈을 때 어느 경로가 더 빨리 보이는가
- Redis 경로가 `miss-fill` 에서 `hit` 으로 바뀌며 체감 속도가 어떻게 달라지는가

## 화면 구성

### 1. Hero

- 대시보드 제목과 현재 목적을 짧게 설명한다.
- 우측에 `redis backend`, `api target` 을 표시한다.
- 사용자는 지금 로컬 모드인지, 프록시 모드인지 상단에서 바로 확인할 수 있다.

### 2. Race Setup

- 검색어
- 지역
- 카테고리
- `동시 비교 실행`
- `메트릭 새로고침`

사용자는 검색 조건 하나만 입력하면 된다.
버튼을 누르면 MongoDB 와 Redis 요청이 같은 시점에 같이 출발한다.

### 3. Winner Panel

- 현재 승자
- 두 경로의 진행 바
- 몇 ms 차이로 먼저 도착했는지 요약

이 섹션은 숫자보다 먼저 “누가 이겼는가”를 보여주는 역할을 한다.

### 4. MongoDB Direct Lane

- 브라우저 체감 latency
- 서버 trace latency
- 결과 수
- cache status
- trace 요약
- 검색 결과 카드 목록

이 섹션은 DB direct 경로가 어떤 응답을 보냈는지 보여준다.

### 5. Mini Redis Cache Lane

- 브라우저 체감 latency
- 서버 trace latency
- 결과 수
- cache status
- trace 요약
- 검색 결과 카드 목록

이 섹션은 Redis cache 경로의 응답을 보여준다.
첫 실행에서는 `miss-fill`, 이후 반복 실행에서는 `hit` 이 보이는 흐름을 확인할 수 있다.

### 6. Service Snapshot

- request count
- error count
- hit ratio
- requests/sec
- avg latency
- cpu
- tx/rx

간단한 운영 상태를 한눈에 확인하는 보조 영역이다.

## 사용자 흐름

1. 검색 조건 입력
2. `동시 비교 실행` 클릭
3. MongoDB 와 Redis 요청을 동시에 전송
4. 먼저 끝난 쪽이 시각적으로 강조됨
5. 두 경로의 trace 와 결과 수를 비교
6. 같은 조건으로 다시 실행해 Redis `hit` 여부 확인
